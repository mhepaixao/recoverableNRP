Abstract

Selecting a set of requirements to be included in the next software release, which has become to be known as the Next Release Problem, is an important issue in the iterative and incremental software development model. Since software development is performed under a dynamic environment, some requirements aspects are highly subject to uncertainties, which should be taken into account when solving this problem through a search technique. Current robust approaches for dealing with these uncertainties are very conservative, since they perform the selection of the requirements considering all possible uncertainty realizations. Thereby, this paper presents an evolution of this robust model, exploiting the recoverable robust optimization framework, which is capable of producing recoverable solutions for the Next Release Problem. Several experiments were performed over synthetic and real-world instances, with all results showing that the recovery strategy handles well the conservatism and adds more quality to the robust solutions.

Instances

Synthetic Instances

The synthetic instance set is composed by 7 randomly generated instances. Each instance has 3 scenarios and the requirement importance value can assume an integer between 1 and 10. The instances were generated with different numbers of requirements, ranging from 50 to 200. A random approach was considered in order to define the requirements cost deviance, where the deviance is set as a random number between 0 (exclusive) and 50% of the requirement cost. The synthetic instance name is in the format I_S_R, where R is the number of requirements. The instance I_S_100, for example, has 100 requirements.

Real-world Instances

The real instances were extracted from bug repositories of two big open source projects, Eclipse (a java integrated development environment) and Mozilla (a set of web applications). This strategy for NRP real instance extraction was adapted from the work by Xuan et al. A bug repository is a forum where users (end users, developers, testers, etc) can report bugs related to the project. In this forum, one bug report may be commented by many users. One bug report is mapped to a requirement. The requirement value of a bug is then calculated as the number of users that commented on that particular bug. In addition, the bug severity is mapped to the requirement cost. Both the requirements value and cost are normalized to the 1 to 10 interval. In all cases, each instance was considered to have only one scenario (with a probability equals to 1). Seven instances, composed by the most important requirements, were extracted from each bug repository. The instances contains 50, 70, 100, 120, 150, 170 and 200 requirements, respectively. The requirements cost deviances were adjusted using the same random approach as described early. The real-world instance name is in the format I_Re_P_R, where P represents the project (E for Eclipse and M for Mozilla), and R is the number of requirements. The instance I_Re_E_120, for example, was generated from the eclipse bug repository and has 120 requirements.
